### 剖析：`retry` 与 `retryWhen` 操作符源码与底层流转机制

在 Kotlin Flow 中，重试机制底层高度依赖 `catchImpl` 的异常可见性控制。`catchImpl` 在此扮演了两个核心角色：
1. **数据分发代理**：隐式处理正常数据的收集与向下游的发送。
2. **异常精准过滤器**：确保其返回值仅包含纯粹的上游业务异常。

#### 1. `retry` 操作符：便捷的参数封装 (Wrapper)

```kotlin
public fun <T> Flow<T>.retry(
    retries: Long = Long.MAX_VALUE,
    predicate: suspend (cause: Throwable) -> Boolean = { true }
): Flow<T> {
    require(retries > 0) { "Expected positive amount of retries, but had $retries" }
    return retryWhen { cause, attempt -> attempt < retries && predicate(cause) }
}
```

**机制分析**：`retry` 操作符本质上是一个高阶的参数组装器。它将开发者常用的两个简单诉求（“最大重试次数”与“异常类型断言”）组合拼接为一个符合 `retryWhen` 签名的 Lambda 表达式：`attempt < retries && predicate(cause)`。通过这种封装，有效降低了调用端的认知负荷。其核心的重试控制流均下沉交由 `retryWhen` 实现。

#### 2. `retryWhen` 操作符：核心状态机与控制流转

```kotlin
public fun <T> Flow<T>.retryWhen(predicate: suspend FlowCollector<T>.(cause: Throwable, attempt: Long) -> Boolean): Flow<T> =
    flow {
        var attempt = 0L
        var shallRetry: Boolean
        do {
            shallRetry = false
            val cause = catchImpl(this) // <--- 控制流核心节点
            if (cause != null) {
                if (predicate(cause, attempt)) {
                    shallRetry = true
                    attempt++
                } else {
                    throw cause
                }
            }
        } while (shallRetry)
    }
```

**机制分析**：`retryWhen` 内部通过一个 `do-while` 循环构建了极简的重试状态机。其最巧妙的设计在于 `val cause = catchImpl(this)`：在整个调用期间，上游数据的收集与对下游的 `emit` 调用，全部在 `catchImpl` 内部静默完成。

通过剖析上述代码，可得出控制流转的三种终极结局：

**结局一：执行顺利，无异常抛出 (Happy Path)**
1. 代码进入 `do-while` 循环，`shallRetry` 初始化为 `false`。
2. 执行 `val cause = catchImpl(this)`。在 `catchImpl` 内部，上游正常生产数据并持续转发给下游。
3. 数据发送完毕后，`catchImpl` 正常执行结束，返回 `null`。
4. 此时 `cause == null`，跳过内部的 `if` 块判定。
5. 步入 `while (shallRetry)`，鉴于状态为 `false`，循环正常终止。重试操作符未对常规数据流产生任何阻断或干涉。

**结局二：拦截到越界异常（下游崩溃或取消信号）**
1. 在 `catchImpl(this)` 代理发数据的过程中，若下游（如消费端 UI 渲染）抛出异常，或当前协程作用域被取消（产生 `CancellationException`）。
2. 依据 `catchImpl` 底层的异常可见性规则，该类异常**不会**作为返回值被抛出，而是在 `catchImpl` 内部被直接 `throw`。
3. 这一行为将直接导致 `retryWhen` 的 `do-while` 循环被强制击穿，重试机制宣告失效。此设计符合系统边界逻辑：当下游消费链路断裂或宿主协程被取消时，针对上游源头的任何重试均属无效操作。

**结局三：合法的上游业务异常触发重试判定**
1. 在调用 `catchImpl(this)` 时，上游在生产数据环节（如网络 IO 请求）抛出纯业务异常（如 `IOException`）。
2. `catchImpl` 成功拦截该上游异常，并将其安全地作为返回值赋给 `cause` 变量。
3. 此时 `cause != null` 成立，进而调用传入的 `predicate` 断言闭包（校验重试次数上限及具体的异常类型）。
   * **若断言拒绝 (返回 `false`)**：直接 `throw cause`，将异常向上层作用域抛出，终止后续执行。
   * **若断言允许 (返回 `true`)**：更新内部状态 `shallRetry = true` 以及尝试次数 `attempt++`。
4. 步入 `while (shallRetry)`，循环条件成立，再次进入 `do` 块起始位置。
5. **重试的底层本质**：系统将再次执行 `catchImpl(this)`。鉴于 Flow 的冷流特性，该调用意味着上游的 `flow { ... }` 构建闭包将被重新触发，从第一行代码开始执行完整的数据生产流，从而实现真正意义上的逻辑重置与重试。
