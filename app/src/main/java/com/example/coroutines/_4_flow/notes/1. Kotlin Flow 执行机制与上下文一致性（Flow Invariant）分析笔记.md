## 1. Kotlin Flow 执行机制与上下文一致性（Flow Invariant）分析笔记

### 一、 核心执行抽象模型：延续体回调与上下文代入
Flow 的底层运行机制绝非基于物理队列的数据生产与消费，而是基于高阶函数与延续体（Continuation）的同步回调。

1. **`flow { ... }` 闭包的本质**：构建流时，大括号内的代码并未执行。它在内存中被封装为一个挂起函数（`suspend FlowCollector<T>.() -> Unit`），定义了数据发射的控制流模板。
2. **`collect { ... }` 闭包的本质**：定义了消费者处理每一个流元素的具体行为逻辑。
3. **`emit` 的代入模型**：当调用 `collect` 触发流的执行后，控制流在 `flow` 闭包内顺序推进。每当执行到 `emit(x)` 时，控制流会暂停当前代码块的推进，并携带元素 `x`，在 `emit` 调用的**当前行、当前协程上下文中**，同步且就地（In-place）执行一次 `collect` 闭包。
4. **上下文跟随特性**：基于上述代入模型，`collect` 闭包的运行时协程上下文（Job 与 Dispatcher），永远严格等同于触发它的那次 `emit` 所在的协程上下文。

### 二、 运行时安全机制：SafeCollector 与并发拦截
普通的 `FlowCollector` 是非线程安全的（Not thread-safe）。若允许并发发射，`collect` 闭包将被多个协程并发执行，导致严重的数据竞争。为此，Flow 引入了 `SafeCollector` 进行严格的运行时状态机校验。

1. **基准上下文绑定**：当发起 `numFlow.collect` 时，底层将其包装为 `SafeCollector`，并永久记录当前发起调用的初始协程 `Job` 作为合法基准。
2. **上下文守卫规则（放行条件）**：每次调用 `emit` 时，`SafeCollector` 将验证当前执行上下文：
    * **规则 A**：当前执行 `emit` 的协程 Job 等于初始基准 Job。
    * **规则 B**：当前执行 `emit` 的协程是初始基准 Job 的**严格同步子协程**（如使用 `coroutineScope` 创建）。此时父协程处于强挂起（阻塞）状态，不存在并发调用的可能，判定为安全。
3. **违规拦截规则（拦截条件）**：
    * 若当前 `emit` 处于由 `launch` 或 `async` 创建的异步并发协程中，`SafeCollector` 判定其破坏了 Flow 的单向顺序执行约束（Flow Invariant），将直接抛出 `IllegalStateException` 并终止程序。

### 三、 代码执行流时序拆解 (基于运行日志)

以下为测试代码的具体并发执行时间线与状态流转分析：

#### 阶段 1：收集器初始化与基准绑定
* **执行逻辑**：外部调用 `scope.launch { numFlow.collect { ... } }`。
* **状态流转**：系统启动新协程（基准 Job：`@2a48dfd5`）。`SafeCollector` 实例被创建并绑定该 Job 地址。
* **日志印证**：`collect job?1: StandaloneCoroutine{Active}@2a48dfd5`

#### 阶段 2：初始发射与上下文一致性验证 (emit 0)
* **执行逻辑**：控制流进入 `flow` 闭包，单线程顺序执行到 `emit(0)`。
* **状态流转**：当前所在协程为基准协程 `@2a48dfd5`。`SafeCollector` 校验通过。`collect` 闭包在当前上下文中被同步回调。
* **日志印证**：`collect job?2: StandaloneCoroutine{Active}@2a48dfd5`（证实 `collect` 闭包运行环境未改变）。成功输出 `collected num: 0`。

#### 阶段 3：异步并发分支的创建 (launch -> emit 1)
* **执行逻辑**：遇到 `launch { emit(1) }`。该 `launch` 通过词法作用域隐式捕获了最外层 `runBlocking` 的 `CoroutineScope`。
* **状态流转**：系统在后台调度器中创建了一个全新的并发协程（Job：`@1f57539`）。主控制流非阻塞地继续向下执行。

#### 阶段 4：合法的同步子域发射 (coroutineScope -> emit 2)
* **执行逻辑**：主控制流到达 `coroutineScope { ... }` 挂起函数。
* **状态流转**：基准父协程 `@2a48dfd5` 被安全挂起。系统分配同步子作用域 `ScopeCoroutine{Active}@58431f17` 执行其内部代码。
* **验证逻辑**：子协程 `@58431f17` 内部执行到 `emit(2)`。`SafeCollector` 向上溯源 Job 树，确认其为基准协程的合法同步子级，未破坏并发安全，予以放行。
* **日志印证**：
    * `emit2 in job: ScopeCoroutine{Active}@58431f17`
    * `collect job?2: ScopeCoroutine{Active}@58431f17`（再次证实：`collect` 闭包的执行上下文精准跟随了合法的 `emit` 所在上下文）。
    * 成功输出 `collected num: 2`。

#### 阶段 5：运行时拦截与程序崩溃 (违规执行 emit 1)
* **执行逻辑**：后台并发协程 `@1f57539` 获取到 CPU 时间片，执行闭包内部的 `emit(1)`。
* **日志印证**：`emit1 in job: StandaloneCoroutine{Active}@1f57539`
* **状态流转**：`SafeCollector` 拦截此操作，核对当前协程 `@1f57539` 与基准协程 `@2a48dfd5` 无任何同步从属关系。
* **抛出异常**：判定为跨协程并发发射。抛出 `IllegalStateException: Flow invariant is violated`，标明预期的 Job 与实际侦测到的违规 Job 地址，主线程随之崩溃，后续流程（如 `emit(3)`）中止。
