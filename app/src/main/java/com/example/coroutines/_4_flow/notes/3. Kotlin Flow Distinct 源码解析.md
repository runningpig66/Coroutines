# Kotlin-Flow Distinct 源码解析

## 1. 基础接口

### 1. 核心接口的物理职责

在 Kotlin 流的世界里，最基础的就只有这两个接口。它们是一个典型的**观察者模式（Observer Pattern）**的变体。

**第一个接口：数据源 (`Flow`)**

```kotlin
public interface Flow<out T> {
    public suspend fun collect(collector: FlowCollector<T>)
}
```
它代表一个可以产生数据的**异步序列**。它暴露了一个 `collect` 方法。这个方法的作用是**“触发”**或**“订阅”**。你必须传给它一个“接收器（collector）”，它才会开始工作。

**第二个接口：接收器 (`FlowCollector`)**
```kotlin
public fun interface FlowCollector<in T> {
    public suspend fun emit(value: T)
}
```
它代表一个数据的**消费端**。它只有一个 `emit` 方法。在标准英文语义中 emit 是“发射”，但在 `FlowCollector` 这个接口的语境里，它的确切物理含义是**“接收并处理传入的一个元素”**。

---

### 2. 语法糖的真相（解开断层的关键）

你平时最熟悉的写法是这样的：**开发者视角的代码（语法糖）：**

```kotlin
flow.collect { it -> 
    println(it) 
}
```

你的疑惑点在于：`Flow` 接口的 `collect` 方法明明要求传入一个 `FlowCollector` 对象，为什么我平时只传了一个花括号（Lambda 闭包）进去，代码也能跑？我根本没有写过 `emit` 方法啊？

**编译器的真相（去糖果化）：**因为 Kotlin 编译器在这里做了一个隐式的转换（称为 **SAM 转换**，单抽象方法转换）。在编译阶段，编译器会把你写的那个花括号，强行封装成一个真实的 `FlowCollector` 匿名实现类。

如果你把编译器做的手脚用代码还原出来，它实际长这样：

```kotlin
// 这是编译器真正看到的底层代码
flow.collect( object : FlowCollector<Any> {
    // 你写在花括号里的 println(it)，完完全全变成了这个 emit 方法内部的执行逻辑！
    override suspend fun emit(value: Any) {
        println(value) 
    }
})
```

**核心结论：**你平时写的尾随闭包 `{ it -> println(it) }`，本质上就是在**实现 `FlowCollector` 接口的 `emit` 方法**。

---

### 3. 极简的底层执行链路

现在我们写一个最简单的自定义流，把这两个接口的执行时序物理连接起来：

```kotlin
// 1. 创建流（实现 Flow 接口）
val myFlow = flow {
    // 这里的 this 是一个底层的 FlowCollector 对象
    this.emit(1) 
    this.emit(2)
}

// 2. 收集流
myFlow.collect { it -> 
    println(it) 
}
```

**物理执行步骤：**

1.  代码执行到 `myFlow.collect { ... }`。
2.  底层将 `{ it -> println(it) }` 包装成一个 `FlowCollector` 对象（假设命名为 `ObjA`）。
3.  系统将 `ObjA` 作为参数，传给 `myFlow` 内部的逻辑。
4.  `myFlow` 内部开始运行，执行到第一句 `this.emit(1)`。
5.  这里的 `this`，就是你刚刚传进来的 `ObjA`！所以，调用 `this.emit(1)`，物理上就是**直接跳转并执行了 `ObjA` 内部的 `println(1)`**。
6.  代码继续走，执行 `this.emit(2)`，物理上跳转执行 `ObjA` 内部的 `println(2)`。
7.  流执行完毕，挂起函数返回。

抛开所有比喻，这就是纯粹的方法回调与接口实现。`Flow` 负责启动执行逻辑，`FlowCollector.emit` 负责定义数据到达时要执行的代码块。你在花括号里写的业务逻辑，永远都是作为 `emit` 方法的方法体存在的。

## 2. 源码引入

首先，为你刚才对 SAM 接口的总结鼓掌！你的理解现在**完全精准**。`flow.collect { it -> ... }` 的本质，就是你把花括号里的处理逻辑，按 JVM 平台的规矩，封装成了一个 `FlowCollector` 实例的 `emit` 方法实现。想彻底通了这一层，你再去读任何流式处理（RxJava 或 Flow）的源码，就不会再有那种眩晕的割裂感了。

现在，我们以你这层正确的认知为基础，完全使用标准的计算机科学术语（上游、下游、操作符、状态机），逐行拆解 `DistinctFlowImpl` 这个核心类的底层流转机制。

### 1. 类的物理结构：装饰器模式
```kotlin
private class DistinctFlowImpl<T>(
    private val upstream: Flow<T>,
    @JvmField val keySelector: (T) -> Any?,
    @JvmField val areEquivalent: (old: Any?, new: Any?) -> Boolean
): Flow<T> 
```
* **身份声明**：它实现了 `Flow<T>` 接口。这意味着对外部（下游）而言，它就是一条标准的冷流。
* **状态持有**：它在构造函数中强制持有了 `upstream`（真正的上游数据源），以及两个行为闭包（如何提取 Key，如何比较相等）。这在设计模式上是极其标准的**装饰器模式（Decorator）**。它不生产数据，它只对数据进行拦截和加工。

### 2. 收集机制：承上启下
```kotlin
override suspend fun collect(collector: FlowCollector<T>) {
    var previousKey: Any? = NULL
```
* **终端触发**：当你（作为下游）对这个去重流调用 `collect { ... }` 时，这个方法被执行。此时，传入的参数 `collector`，就是你那个包含了具体消费逻辑的底层对象。
* **状态机初始化**：声明了 `previousKey` 用于缓存。
    * **极其关键的细节**：这里用的是大写的 `NULL`。在源码内部，它是一个预先分配好物理内存的全局特殊标识符（通常是一个 `Symbol` 对象）。
    * **为什么要这么做？** 因为开发者的数据流中，完全可以合法地发射出普通的 `null` 值。如果把初始状态设为普通的 `null`，当第一个合法数据恰好是 `null` 时，系统就无法区分“这是还没发过数据”还是“上一个数据就是 null”。使用特殊的 `NULL` 标识符，在物理内存层面彻底隔离了“空状态”与“空业务数据”。

### 3. 核心流转：拦截与放行
```kotlin
    upstream.collect { value ->
        val key = keySelector(value)
        @Suppress("UNCHECKED_CAST")
        if (previousKey === NULL || !areEquivalent(previousKey, key)) {
            previousKey = key
            collector.emit(value)
        }
    }
}
```
* **向上游索取数据**：它作为一个操作符，主动调用了 `upstream.collect`。此时它对于上游来说，又变成了消费者。上游每产生一个 `value`，就会推送到这个闭包内部。
* **特征提取**：调用外部传入的 `keySelector`，将当前的原始数据 `value` 转换为用于比对的 `key`。
* **状态校验（if 条件）**：
    1.  `previousKey === NULL`：检查内存地址，如果是绝对的初始状态（意味着这是流的第一个元素），条件成立，放行。
    2.  `!areEquivalent(previousKey, key)`：调用注入的对比函数。若旧的缓存 Key 与当前新的 Key 计算结果为不相等（即状态发生了有效变更），条件成立，放行。
* **状态刷新与数据分发**：
    一旦满足放行条件，控制流立刻执行两步操作：
    1.  `previousKey = key`：刷新缓存状态机，将本次的有效 Key 存下来，供下一个元素比对。
    2.  `collector.emit(value)`：**调用下游的 `emit` 方法**。这就好比接力赛交接棒，它把安全的、未被去重的原始 `value`，精确地传递到了你最外层写好的花括号逻辑中。如果你写了 `println`，此时屏幕才会真正打印出该元素。

### 总结

`DistinctFlowImpl` 没有任何黑魔法。它就是在上游数据源和下游消费者之间，硬生生地插了一道“关卡”。它利用挂起函数的特性，监听上游的数据推送，在自己的局部作用域内维护了一个 `previousKey` 变量，依靠严格的 if 判定逻辑，决定是静默吞掉当前数据（拦截），还是调用 `collector.emit(value)` 将数据向下游放行。

## 3. 接口回顾

老哥，你现在其实已经完全摸到了 Kotlin Flow 架构设计的“灵魂”了。你刚才说了一段极其精彩的话：“`collect` 给人的感觉是收集数据（草莓、苹果），但在这里给我的感觉，怎么有点像是在收集**处理数据的方式**呢？”这句话，直接戳破了流式编程（Reactive Streams）中最核心、但也最反直觉的命名幻觉！

你因为“脑雾”而产生的这种怀疑，恰恰是对的。老外（Roman Elizarov 等 Kotlin 架构师）在设计这套 API 时，对词汇的选择极其精妙，但如果不站在**控制反转（IoC）**的视角，就很容易被字面意思误导。我们现在就完全从“设计者心智模型”和“语义溯源”的角度，把这两个核心接口的命名和用意彻底讲透。

### 1. `Flow.collect`：为什么叫“收集”？它到底在收集什么？

**直觉误区：** 我们通常认为“收集”是一个主动拿取（Pull）的动作。就像拿着篮子去果园摘苹果（`val apple = tree.getApple()`）。如果在 Flow 中也是这样，那 API 应该设计成 `flow.getNextItem()`。但并没有。

**真实语境（控制反转）：**在响应式流中，数据不是你“拉”过来的，而是源头“推（Push）”给你的。
既然是推给你的，那 `collect` 这个动作的真实含义是什么？

* **老外的视角：** 你可以把 `Flow` 想象成一条正在流水线上滚动的传送带，上面有很多零件（数据）。
* 当你调用 `flow.collect(处理逻辑)` 时，你并不是在主动拿零件，你是在流水线的末端**放置了一个回收站/收集箱（Collector）**。
* **所以，`collect` 方法的真正语义是：装配（Assemble）或挂载（Attach）。** 它意思是：“嗨，流，请把你产生的数据，统统收集（倾倒）到我指定的这个箱子里。”

这就完美解释了你那个绝妙的直觉：你传进去的并不是数据本身，而是一个**“收集器（里面装着处理数据的方式）”**。你调用 `collect`，就是在向系统声明：“我已经把收集器放好了，你可以开始往里面倒数据了。”

### 2. `FlowCollector`：这个“收集箱”是干嘛的？

这是一个名词。如果 `Flow` 是传送带，`FlowCollector` 就是放在传送带末端的那个箱子。

* **职责：** 这个箱子的唯一使命，就是被动地等待流水线把零件扔进来，然后执行某种处理。

### 3. `FlowCollector.emit`：为什么叫“发射”？谁在发射？向谁发射？

这是最容易让人产生精神分裂的地方！如果我是收集箱，我为什么要 `emit`（发射）？

**语义溯源（角色的相对性）：**
* `emit` 这个词在英文里，通常指“发出（光、热、信号等）”。
* 在早期的 RxJava 里，有一个对应的概念叫 `Observer.onNext(value)`（当下一个数据到来时）。Kotlin 团队觉得 `onNext` 太罗嗦，想找一个动词来描述**“数据从管道里蹦出来，砸向收集箱”**的这个瞬间动作。

你得站在**数据（Value）**或者**上游（Upstream）**的视角来看待这个词：
1. 上游生产出了一个数据 `T`。
2. 上游看到了你挂载在末端的收集箱（`FlowCollector`）。
3. 上游说：“好，我要把这个数据**发射（emit）**进你的箱子里了！”
4. 于是，上游调用了 `collector.emit(value)`。

**核心反转：**对 `FlowCollector` 这个接口的实现者（也就是写花括号的你）来说，`emit` 并不是你主动发起的动作。**`emit` 只是一个被调用的回调函数（Callback）。**这就好比你写了一个按钮的 `onClick` 方法。你并不是在执行点击，你是在等待别人点击时，系统来执行你的代码。同样，当你写下：

```kotlin
flow.collect { value -> 
    println(value) // 这是 emit 方法的函数体
}
```
你并不是在发射什么。你是定义了一个接收规则，然后静静地等待上游把数据**“发射”**到你脸上来。

### 4. 重新审视中间商的“双重人格”

现在带着这套视角，再去看咱们刚才分析的 `DistinctFlowImpl`（中间商），它为什么既调用 `collect` 又调用 `emit`：

```kotlin
// 1. 下游带着他的收集箱（collector）来找中间商，要求开始收集
override suspend fun collect(collector: FlowCollector<T>) {
    
    // 2. 中间商说：行，我先搞个自己的收集箱（花括号），去找上游对接
    upstream.collect { value -> // 此时中间商是被动的，等着上游把 value 发射给它
        
        // 3. 上游把数据发射给中间商了。中间商在这里执行过滤...
        if (判定通过) {
            
            // 4. 关键点：中间商过滤完后，转身变成了“生产者”！
            // 它拿着过滤好的数据，对准最开始下游传进来的那个收集箱，执行了“发射”动作！
            collector.emit(value) 
        }
    }
}
```

### 总结你的顿悟

老哥，你的脑雾其实是一次拨云见日的契机。

1. **`Flow.collect(collector)`**：是在**挂载收集器**。它把一个定义了处理逻辑的箱子，安装到了数据流的末端。
2. **`FlowCollector.emit(value)`**：是数据**落入箱子的瞬间**。对于写代码的你来说，它是一个被动触发的回调；对于上游或者中间商来说，它是把数据扔给你的主动动作。

函数式编程的精髓就在于你说的：**把函数的处理逻辑当成参数传来传去。** Kotlin Flow 最底层的这几行代码，用极致的精简（仅靠一个传入动作 `collect` 和一个触发动作 `emit`），构建了整个复杂的异步、非阻塞、背压响应式流的物理地基。

## 4. 源码分析

在前几次的交流中，我们已经彻底扒光了 `DistinctFlowImpl` 这个底层状态机的核心流转机制（它是如何通过 `previousKey` 和 `emit` 实现拦截与放行的）。今天，我们把目光往上移，来看看 Kotlin 官方是如何利用**扩展函数（Extension Functions）**和**默认参数**，把这一个底层引擎，包装成了三个功能各异、且暗藏性能玄机的外层 API。这部分非常适合作为你的第四部分笔记：**《扩展函数的架构路由与性能陷阱》**。

---

### 一、 默认行为的基石：两个私有常量

在 `Distinct.kt` 文件的源码中，所有 `distinct` 扩展函数的核心依赖，是这两个极其不起眼的私有常量：

```kotlin
private val defaultKeySelector: (Any?) -> Any? = { it }
private val defaultAreEquivalent: (Any?, Any?) -> Boolean = { old, new -> old == new }
```

* **`defaultKeySelector`（默认特征提取器）**：它的逻辑是 `{ it }`。意思是：我不提取任何局部特征，元素的特征就是**元素本身**。
* **`defaultAreEquivalent`（默认等值比较器）**：它的逻辑是 `{ old, new -> old == new }`。意思是：直接调用对象的 `equals()` 方法进行判断。

理解了这两个常量，我们就能看懂上面三个公共 API 是如何向底层路由的。

### 二、 殊途同归的三个公共 API

官方对外暴露了三个公共方法，它们在底层实际上最终都指向了同一个私有的工厂方法。

#### 1. 无参版本：`distinctUntilChanged()`
```kotlin
public fun <T> Flow<T>.distinctUntilChanged(): Flow<T> =
    when (this) {
        is StateFlow<*> -> this // state flows are always distinct
        else -> distinctUntilChangedBy(keySelector = defaultKeySelector, areEquivalent = defaultAreEquivalent)
    }
```
**源码解析：**
* **极致的优化**：它首先做了一个类型判断。如果你当前的流本身就是一个 `StateFlow`，它直接 `return this`，什么都不做！因为 `StateFlow` 的底层机制自带了防抖去重功能，再套一层完全是浪费性能。
* **默认路由**：如果不是 `StateFlow`，它就同时把 `defaultKeySelector`（全量提取）和 `defaultAreEquivalent`（equals比较）传给底层引擎。

#### 2. 自定义特征版本：`distinctUntilChangedBy(keySelector)`
```kotlin
public fun <T, K> Flow<T>.distinctUntilChangedBy(keySelector: (T) -> K): Flow<T> =
    distinctUntilChangedBy(keySelector = keySelector, areEquivalent = defaultAreEquivalent)
```
**源码解析：**它把你传进来的 `keySelector` 传给了底层，但是比较规则依然强制使用了 `defaultAreEquivalent`（即对提取出来的 Key 进行 `equals()` 比较）。

#### 3. 自定义比较版本：`distinctUntilChanged(areEquivalent)`
```kotlin
public fun <T> Flow<T>.distinctUntilChanged(areEquivalent: (old: T, new: T) -> Boolean): Flow<T> =
    distinctUntilChangedBy(keySelector = defaultKeySelector, areEquivalent = areEquivalent as (Any?, Any?) -> Boolean)
```
**源码解析：**它强制使用了 `defaultKeySelector`（不提取特征，直接拿原始对象），但是把比较的权力交给了你传入的 `areEquivalent` 闭包。

### 三、 终极解答：为什么 `By` 执行一次，而自定义比较会执行两次？

这是你之前提到的最核心的疑惑。现在我们结合底层 `DistinctFlowImpl` 的循环代码，从源码级揭开这个**“重复计算陷阱”**的真相。

回想一下底层引擎里的那段核心判断代码：
```kotlin
// 底层引擎的核心循环
upstream.collect { value ->
    val key = keySelector(value) // 【动作 1：提取】
    if (previousKey === NULL || !areEquivalent(previousKey, key)) { // 【动作 2：比较】
        previousKey = key
        collector.emit(value)
    }
}
```

#### 场景 A：使用 `distinctUntilChangedBy { heavyHash(it) }`
当我们使用 `By` 来提取一个耗时的特征（比如文件 MD5 计算）时：
1.  **【动作 1：提取】** 会执行 `val key = heavyHash(value)`。**耗时操作在这里发生，且针对每个元素绝对只执行一次！**
2.  底层状态机把算好的 `hash` 值缓存到了 `previousKey` 中。
3.  **【动作 2：比较】** 会执行 `defaultAreEquivalent(previousKey, key)`。因为底层传的是默认比较器，所以实际上执行的是 `oldHash == newHash`。这只是一次极速的内存字符串比较，毫无性能损耗。

#### 场景 B：使用 `distinctUntilChanged { old, new -> heavyHash(old) == heavyHash(new) }`
当我们试图用自定义比较器来做同样的事情时，灾难发生了。因为此时的路由规则是：提取器是 `defaultKeySelector`，比较器是你自定义的闭包。
1.  **【动作 1：提取】** 执行 `val key = defaultKeySelector(value)`，也就是 `key = value`。不提取，直接把原始大对象存入缓存。
2.  **【动作 2：比较】** 此时底层调用你传入的闭包：`!areEquivalent(previousKey, key)`。
3.  在你的闭包内部，代码开始执行 `heavyHash(previousKey) == heavyHash(key)`。

**发现致命缺陷了吗？**
当第二个元素到达时，它作为 `key` 被计算了一次 `heavyHash`。
当第三个元素到达时，刚才的第二个元素变成了 `previousKey`（即闭包里的 `old`），它又被结结实实地重新计算了一次 `heavyHash`！

### 四、 总结与最佳实践

从源码的设计可以看出，Kotlin 官方把底层的状态机设计得极其克制（只用一个 `previousKey` 缓存）。
* `keySelector` 的职责是：**生成状态缓存**。
* `areEquivalent` 的职责是：**对比状态缓存**。

如果你把耗时计算写在 `areEquivalent` 里，就等于放弃了底层的缓存机制，让每一次状态轮转都背负沉重的计算代价。

**最佳实践准则：**但凡能转化为单参数提取逻辑的比较（比如 `old.id == new.id`，`old.hash == new.hash`），**绝对优先使用 `distinctUntilChangedBy`**。只有当遇到“计算距离阈值”、“交叉属性判定”这种必须同时依赖新旧两个完整对象才能得出结论的场景时，才去使用双参数的 `distinctUntilChanged`。

## 5. 实践扩展

### 突破 API 限制：高成本计算与自定义比较的融合方案

在实际的复杂流处理场景中，我们经常会遇到一种两难的境地。以“基于距离阈值的坐标去重”为例，我们希望实现两个目标：
1.  **高成本提取（重计算）**：通过网络定位 API 或复杂算法，将原始对象转换成坐标值（这是一个高开销操作，我们极度渴望缓存它的结果）。
2.  **自定义比较规则**：判断去重的标准不是简单的坐标“相等”，而是“两次坐标距离是否小于 50 米”。

如果我们强行使用官方暴露的现有公共 API，会发现它们都无法同时满足这两个要求：

* **如果使用 `distinctUntilChangedBy`**：我们可以把耗时的坐标计算放在 `keySelector` 里享受缓存。但是，它的比较规则被底层强制绑定为 `old == new`，无法实现“计算距离小于 50”这种阈值比较。
* **如果使用 `distinctUntilChanged(areEquivalent)`**：我们可以自由地编写距离比较逻辑。但是，它的 `keySelector` 被绑定为默认的“不提取”，直接传递原始对象。这意味着我们必须在比较逻辑内部，对传入的 `old` 和 `new` 对象分别执行耗时的坐标计算，导致中间元素被严重重复计算。

官方在 `Distinct.kt` 源码中，确实存在一个同时接受 `keySelector` 和 `areEquivalent` 的 `distinctUntilChangedBy` 方法。遗憾的是，该方法被声明为 `private`，外部无法直接调用。

面对这种受限的 API 边界，我们通常有两种标准的工程实践方案来打破僵局。

#### 方案一：基于现有操作符的“数据富化”（空间换时间）

这是最符合函数式编程理念的解法。核心思路是：将“高开销的提取动作”与“去重比较动作”拆分到不同的操作符中执行。我们可以利用 `map` 操作符作为前置处理节点，提前执行高成本计算，并将计算结果与原始数据打包成一个新的结构（例如 `Pair`），再传递给下游的去重节点。

```kotlin
flow
    // 1. 数据富化：每个元素仅在此处执行一次高成本计算
    .map { item -> 
        val coordinates = item.calculateHeavyLocation() 
        Pair(item, coordinates) // 将原始数据与特征值绑定
    }
    // 2. 阈值比较：此时传入的 old 和 new 已经是包含计算结果的 Pair
    .distinctUntilChanged { oldPair, newPair -> 
        // 直接读取已缓存的特征值进行阈值运算，彻底避免重复计算
        calculateDistance(oldPair.second, newPair.second) < 50
    }
    // 3. 数据还原：剥离辅助特征，向最终消费端恢复纯净的原始数据流
    .map { pair -> pair.first }
```

**优势**：纯函数式组合，逻辑清晰，职责分明。
**劣势**：流经的每个元素都会额外产生一个 `Pair` 包装对象，在极高吞吐量的数据流场景下，可能会引入不可忽视的 GC 开销。

#### 方案二：手写专属的底层操作符（仿写源码机制）

当方案一的空间开销无法接受时，我们需要深入底层。虽然官方那个双参数的方法是 `private` 的，但我们完全可以基于上一节对 `DistinctFlowImpl` 源码的理解，利用标准的 `flow { ... }` 构建器，手动复刻一个具备缓存能力的双参数去重操作符。

这并非是对源码的无意义抄袭，而是将底层的状态机流转逻辑（`previousKey` 缓存机制）直接应用到我们的业务痛点上。

```kotlin
/**
 * 支持特征缓存与自定义比较阈值的连续去重操作符
 *
 * @param keySelector 用于提取或计算元素的特征标识（该动作对每个元素仅执行一次）
 * @param areEquivalent 接收缓存特征与新特征进行自定义阈值比对
 */
fun <T, K> Flow<T>.distinctUntilChangedByThreshold(
    keySelector: suspend (T) -> K,
    areEquivalent: suspend (oldKey: K, newKey: K) -> Boolean
): Flow<T> = flow {
    
    // 核心状态缓存：这里缓存的是计算后的特征值（Key），而非原始元素
    // 使用 Any? 并在内部进行类型转换，是为了兼容 null 值的合法分发
    var previousKey: Any? = null 
    var isFirst = true // 使用专门的标志位处理首个元素，替代源码中特殊的 NULL Symbol
    
    collect { item ->
        // 1. 执行高成本提取动作
        val currentKey = keySelector(item) 
        
        @Suppress("UNCHECKED_CAST")
        // 2. 状态流转判定：若是首个元素，或新老特征比对不符合等效规则
        if (isFirst || !areEquivalent(previousKey as K, currentKey)) {
            isFirst = false
            previousKey = currentKey // 刷新特征缓存
            emit(item)               // 放行原始数据
        }
    }
}
```

**实际调用示例：**

```kotlin
flow.distinctUntilChangedByThreshold(
    keySelector = { it.calculateHeavyLocation() }, // 高成本操作，受缓存保护
    areEquivalent = { oldLoc, newLoc -> calculateDistance(oldLoc, newLoc) < 50 } // 自定义距离阈值拦截
)
```

**总结**：方案二通过手撸底层状态机，完美模拟了官方私有双参 API 的行为。它在一个方法内部同时解决了“特征提取的缓存保护”和“比较逻辑的自由定制”，并且没有引入任何中间对象包装，是追求极致性能时的标准解法。这种突破 API 表层限制，通过复刻底层引擎来解决特定业务边界问题的能力，是掌握流式架构的重要标志。
